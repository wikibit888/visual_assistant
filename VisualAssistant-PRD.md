> 最后更新：2026-06-13
>
> **本版四项基线决策（决策人 2026-06-13 拍板，逐条钉死）：**
> 1. **编排形态 = 工具调用 agent 循环（反应式）**：每个用户回合，编排器（planner LLM）看到工具清单后自主多步规划——要么直接出文本回答，要么发起若干轮工具调用（视觉 / 天气 / 记忆）。**取代** v0.4 的固定五态路由 + zero-shot 意图分类器。反应式 = 被动等用户回合，agent 自身不主动发起动作（唯一的"主动"是端侧姿态守护这条独立的确定性护栏，非 agent 自发）。
> 2. **交付框定 = 仍是单人 36h 现场 demo**：保留止损线、S3 唯一牺牲位、彩排神圣性。**知情风险**：agentic 非确定性与"可彩排的确定性"天然冲突，M5「连续 3 次无致命失误」难度高于写死流水线。**对冲机制**：见下方"设计脊梁"，并以 M2 硬门 + rails mode 兜底（§10）。
> 3. **改写深度 = 产品 + 架构同改**：§7 重写为 agent 编排器 + 工具注册表 + 工作记忆层；契约八（意图分类路由）整条替换，契约三 / 四 相应变更，新增契约十（工作记忆）。变更清单见 §7.7，需同步到 `contracts/` 与 `CLAUDE.md`（决策人已授权）。
> 4. **场景范围 = 全开放学习 + 生活伙伴**：作业辅导为核心锚点（数学方程为彩排锚题），但同一 agent 框架自然覆盖坐姿守护、穿搭建议、闲聊与任意临场请求。能力面开放，**彩排保障面仍是 S1/S2/S3 + 诚实兜底的开放挑战**。
>
> **设计脊梁（贯穿全文）——「planner 提议，确定性护栏裁决」：**
> agent 在"怎么帮"上自由（自主选工具、自主定提示粒度、自主决定看不看）；但产品立场与演示安全做成 **agent 改不动的确定性代码**：答案拦截、置信门控、澄清上限、姿态不抢话、视觉预算、loop 上限。**护栏在编排循环之外、不可被 planner 覆盖**。这是把"agentic 自主"与"36h demo 确定性"对冲掉的唯一办法，也是本产品区别于"放飞的 agent demo"的核心工程纪律。
>
> **继承自上一线的评审教训（已生效，不再复盘）**：S4 物品鉴定整场景删除（墓碑见 §6.3）；答案拦截硬防线（§3.5）；复述确认；批改四值 verdict；对讲机模式升格为正式特性；语音链路止损线（4h/7h/10h）；隐私基线（音视频不留存、姿态端侧）。

## 1. 产品定位与说明

### 1.1 一句话定位

**Visual Assistant 是一个"看得见、听得懂、自己会编排、还管得住自己"的桌面学习生活伙伴**——以学生作业辅导（数学方程为核心锚题）为主场景，通过摄像头与麦克风实时多模态交互。它不是写死的演示流水线，而是一个**反应式 agent**：每个回合自主判断"要不要看、看什么、要不要查天气、给多大提示"，再在确定性护栏内作答。

### 1.2 目标用户

| 用户 | 角色 | 核心诉求 |
|---|---|---|
| 学生（12–15 岁为锚，框架不设限） | 主用户 | 做题卡住时有人**引导**（而非直接给答案）；不被打扰但被照看；任何临场问题都能搭把手 |
| 家长 | 旁观者（**买单者**） | 了解孩子学习卡点与坐姿健康；确认 AI"不代写作业"；相信这个 agent"管得住自己" |

> 差异化定位：市面作业 App 比谁给答案快、谁能力面广；本产品卖的是 **"一个自主但自律的 agent"**——它能力开放（什么都能搭把手），却在最该克制的地方（报答案、冤枉学生、抢话）由硬护栏锁死。买账的是家长与评委；使用者与买单者分离是本产品的前提认知。

### 1.3 产品立场（设计原则）

1. **引导者，不是答案机。**（**agentic 化后权重最高**）数学辅导全程苏格拉底式提问，AI 永不直接报答案。agent 越自由，越不能靠提示词自觉——由**提示词 + 答案拦截硬防线**（§3.5）双层保障，拦截层在编排循环之外。
2. **一直在看，但不打扰。** 坐姿守护后台静默运行；提醒**由确定性护栏插入对话间隙**，不由 agent 自发插话（呼应"反应式"决策）。
3. **先确认，再行动。** 视觉识别结果先复述确认；有歧义先澄清（澄清≤1 次）；**看不清就明说，绝不编造**。
4. **每一帧都要花得值。** 视觉调用按需、识别结果文本化驻留；**工具自由的快路径**（多数引导回合零工具调用，纯文本模型）是最大省钱杠杆。
5. **自主，但可控。**（**本版新增**）agent 的自主性被三道确定性边界约束：每回合 loop 上限（防失控循环）、护栏不可覆盖、超时即维持现场景。**非确定性只允许出现在"语言与策略"，不允许出现在"立场与时序"。**

### 1.4 产品形态：从"写死流水线"到"工具调用 agent"

统一对话入口，编排核心从 v0.4 的「VAD/ASR → 意图分类器 → 分流到固定场景流水线」**改为反应式 agent 循环**：

```
用户语音 ─ 端侧 VAD/ASR ─→ asr.final
                              │
        ┌─────────────────────▼──────────────────────────────────┐
        │           Agent 编排循环（模块 A，反应式）               │
        │  1) 感知  asr.final + 工作记忆(active_problem/focus/…)   │
        │  2) 规划  planner LLM(deepseek-chat) 见工具清单：         │
        │           ├ 无需工具 → 直接出候选回复（**快路径**）       │
        │           └ 需工具   → 发起 ≤N 轮工具调用                 │
        │  3) 行动  dispatch: vision.read_problem / check_draft /  │
        │                     observe / weather.get / memory.*     │
        │  4) 作答  组合工具结果 → 候选回复                         │
        └─────────────────────┬──────────────────────────────────┘
                              │ 候选回复
        ┌─────────────────────▼──────────────────────────────────┐
        │  确定性护栏（agent 不可覆盖）：                            │
        │  答案拦截 · 置信门控 · 澄清上限 · loop 上限 · 视觉预算     │
        └─────────────────────┬──────────────────────────────────┘
                              │ tts.say
                         模块 B 播放
   （模块 D 姿态守护 = 端侧并行，独立发 posture.alert；护栏层在 gap.open 窗口择机插入，
     agent 不参与该决策——这是产品唯一的"主动"，且是确定性护栏而非 agent 自发。）
```

**与 v0.4 的关键差异：**
- **没有独立意图分类器**：路由 = planner 的工具选择 + focus 维持，不再是一次性 zero-shot 分类。
- **没有固定场景流水线**：S1/S2/S3 不再是写死的步骤序列，而是 agent 用工具 + 技能库**临场编排**的结果。
- **粘滞 / 澄清 / 低置信**从"分类器标签"降为"planner 策略 + 确定性护栏"双轨（见 §7.4）。

### 1.5 范围声明

- **开发资源硬约束**：1 名开发者 × 36 小时（AI coding agent 并行开发独立模块；本人负责契约、设备链路调试、集成；**agent 产出按 20% 计提 review/修复工时**）。
- **v0.1 做**：单用户、单设备（带摄像头麦克风的笔记本）、中文语音交互；reactive 工具调用 agent；S1/S2/S3 三个有保障的场景 + **开放能力面（任意临场请求经同一 loop 处理，诚实兜底）**。
- **v0.1 不做**：主动发起式自主（agent 不自发开口，§决策 1）；多步长程任务规划（loop 上限封死）；S4 物品鉴定（墓碑见 §6.3）；账号体系、多用户档案、长期数据存储、移动端原生 App。
- **能力开放 ≠ 保障开放**：agent *能尝试* 任意请求，但 *彩排保障* 仅覆盖 S1/S2/S3 与开放挑战路径的诚实兜底。开放面翻车时以"看不清/帮不上就明说"收口，绝不编造。
- **隐私基线**：音视频流不留存，仅会话内内存中转；工作记忆仅在内存、不落盘；姿态计算全程端侧，原始画面不因坐姿检测上传云端。

## 3. 场景一（S1）：作业辅导 —— 主场景

### 3.1 场景描述

学生在书桌前做作业（彩排锚题为一元一次/二次方程），遇到不会的题，用手指着练习册求助。**agent 自主判断**要不要抓帧识题、什么时候够了可以纯文本引导、什么时候该抓帧批改草稿；通过分步提问引导学生自己解出来；学生写完后请 AI 检查草稿，AI 只定位第一处错误的位置与类型，让学生自己改正。

> 与 v0.4 的差别在"编排"而非"剧情"：v0.4 是写死的 `识题→3 轮引导→批改` 序列；本版是 agent 每回合自己决定下一步，序列是**涌现的**，但被答案拦截与置信门控约束在安全区内。

### 3.2 用户故事与具体 Case

#### P0 —— 核心闭环

| ID | 用户故事 | 验收标准 | 状态 |
|---|---|---|---|
| S1-01 | 我指着练习册某道题说"这道不会"，AI 能准确读出我指的那道题 | agent 自主发起 `vision.read_problem`（整帧多模态理解，不做指尖坐标检测）；识别后**先复述确认**；**澄清上限 1 次**（确认体话术"是第 5 题这道方程吧？"，不用道歉体），第二次直接请用户报题号；**置信度低→护栏拦截，明说"看不清"，绝不猜题开讲**；演示用整本练习册逐题预实测 | 计划 |
| S1-02 | 我希望 AI 一步步提问引导我，而不是直接告诉我答案 | agent 每轮只问一个最小步骤问题；答对推进、答错给更小提示而非纠正；连续受挫时沿**提示粒度阶梯**（方向级→操作级→示范级）下沉，由 planner 策略决定下沉时机；**答案拦截硬防线兜底**：连续三次"我不会"不得报出最终答案（彩排压力测试） | 计划 |
| S1-03 | 我写完解题过程说"你看看"，AI 检查草稿并指出第一处错误 | agent 自主发起 `vision.check_draft`；只定位错误的行与类型，不报正确答案；批改结果走**四值 verdict 契约**（见 §7.7 契约三）；**非高置信一律不播报错误**（置信门控护栏），改播"第 X 行有点看不清，你念念？"；演示主路径用工整字迹 + 预埋移项符号错误；**预授权降级开关**：三次彩排中批改任一次失败，演示日自动切"口述批改"剧本，不再临场争论；口述路径本身须彩排过一次 | 计划 |

### 3.3 关键交互设计

- **指代对齐**：多模态整帧理解 + 复述确认 + 澄清问句（≤1 次）三层兜底。
- **复述确认**：所有识题结果先口头复述，学生确认后才开讲——产品细节兼延迟掩盖（agent 发起视觉工具时，复述话术正好盖住工具往返延迟）。
- **引导粒度阶梯**：方向级（先处理哪一边）→ 操作级（把 2x 移过去）→ 示范级（只示范这一步），逐级下沉，绝不跳到答案。**这是 planner 的可用动作集，不是写死序列**；下沉与否由 agent 临场判断，但"不得到达答案"由 §3.5 护栏锁死。

### 3.4 成本设计

- **识题一次，文本驻留**：识题成功后缓存为工作记忆 `active_problem`，后续引导回合走**零工具的快路径**（纯文本模型）。单题视觉调用 ≤ 3 次（识题 1 + 批改 1–2）。
- **快路径是最大杠杆**：多数引导回合 planner 直接出文本、不调工具 → 单回合 1 次 LLM 调用（比 v0.4"分类器 + 回复"的 2 次更省，见 §7.6）。
- 批改仅在学生明确说"看看"且 agent 判定需要时抓帧，不做持续监看。
- 引导对话历史直接截断（演示 5–7 分钟，不做摘要压缩）。

### 3.5 答案拦截硬防线（继承，权重升高）

产品立场"不报答案"**绝不依赖 agent 自觉**（agent 越自由越要硬拦）：

1. 识题成功时，让模型**顺带解出最终答案**，存入工作记忆 `solved_answer`（**不播报、不入明文日志**）。
2. 此后每条**候选回复在出站前**经过**正则组合拦截**：答案触发词（"x ="、"答案是"、"等于"等）与 `solved_answer` 数值**组合命中**才触发，替换为预置追问话术——纯字符串处理，无 LLM 调用、无延迟成本。
3. 组合匹配规避误杀正常对话中的数字；误杀场景纳入彩排专测。
4. **拦截层位于编排循环之外**：planner 的任何输出都必须穿过它才能到达 TTS。该机制与"禁出答案"判定口径同属契约级（契约九，M0 钉死）。

## 4. 场景二（S2）：坐姿/驼背检测提醒 —— 守护功能

### 4.1 场景描述

学习过程中，端侧持续监测学生坐姿。检测到驼背或头部离桌面过近且持续一段时间后，**由确定性护栏在对话间隙温和提醒**（非 agent 决策）；提醒次数计入学习小结。

### 4.2 用户故事与具体 Case

#### P0

| ID | 用户故事 | 验收标准 | 状态 |
|---|---|---|---|
| S2-01 | 我驼背超过 30 秒时收到一句温和提醒 | 持续 30 秒才触发（防低头写字误报）；固定角度阈值 + 单级预置话术模板随机抽取，无升级状态机；**话术不经 agent / LLM** | 计划 |
| S2-02 | 提醒不打断我的解题思路 | 提醒只在**间隙仲裁**（契约四）广播的 `gap.open` 窗口播出；**提醒是可丢弃事件，不排队**（窗口不足直接丢弃，驼背持续会再次触发）；同类提醒 60s 冷却；持续 20s 无间隙时允许在按句 TTS 的句界插入；**对讲机模式下间隙判定为确定性**（松键且 TTS 结束 ≥1s） | 计划 |

> **与决策 1 的一致性**：S2 提醒是"主动"的，但它来自端侧 D + 确定性间隙仲裁，**不是 LLM agent 的自发开口**。agent 本体仍是纯反应式。这条边界 M0 钉死，防止后续把"提醒"实现成 planner 的一个自由动作而破坏可控性。

#### 已删除（沿用上一线决议）

| ID | 原内容 | 删除理由 |
|---|---|---|
| S2-03 | 提醒升级策略（多级语气） | 状态机复杂；汇报职能并入 S1-06 计数 |
| S2-04 | 连续 40 分钟起身建议 | 演示时长内无法展示 |
| S2-05 | 坐姿趋势周报 | 需长期数据 |

### 6.2 用户故事

## 7. 系统架构与端云协同（agentic 重写）

### 7.1 总体架构与模块重定位

技术栈不变：**Chrome 浏览器页面**（getUserMedia、内建 AEC、MediaPipe JS、vad-web）+ **Python/FastAPI/单 WebSocket 后端**。**逻辑 5 模块、物理 2 进程**，不用 Electron。

模块**名称沿用 A–E 以保持与代码/CLAUDE.md 连续，但角色按 agentic 重定位：**

| 模块 | v0.4 角色 | **v0.1 角色（重定位）** | 物理位置 |
|---|---|---|---|
| **A 编排核心** | 五态路由 + 意图分类 + 仲裁 | **Agent 编排器**：跑工具调用循环（planner LLM）、维护工作记忆、dispatch 工具、**执行确定性护栏**、间隙仲裁、turn_id、成本计数 | `server/a_core/` |
| **B 语音 I/O** | 麦克风/扬声器唯一属主 | **不变**：VAD/PTT/播放队列（前端）+ 云 ASR/TTS 适配器（后端） | `web/src/modules/b_voice.js` + `server/b_voice/` |
| **C 视觉服务** | vision.request→result | **视觉工具实现**：被 agent 当作 `vision.*` 工具调用；重试与预算 | `server/c_vision/` |
| **D 姿态守护** | 端侧角度计算，只发 alert | **不变**：纯端侧，只发 `posture.alert`，绝不出声、绝不入 agent loop | `web/src/modules/d_posture.js` |
| **E 场景流水线** | 纯函数场景流水线 | **技能/提示词库**：苏格拉底引导、穿搭建议、家长摘要等，作为 agent 可调用的技能/工具，**不再是写死流水线**；提示词权属仍归 E | `server/e_scenes/` → 建议更名 `server/e_skills/` |

**铁律（更新）：**
- 跨模块只走总线消息（契约一信封 + 进程内分发），禁止互相 import 内部对象。
- `gap.open` 只能由 A 广播；`tts.say/stop` 只能发给 B；D 的输出只有 `posture.alert`。
- **扬声器上每一个字都必须经过 A 的答案拦截 + 置信门控护栏**（护栏在 planner 之外）。
- **planner 不得绕过护栏直接产生 TTS**；护栏不得被任何提示词关闭。

### 7.2 Agent 编排循环（契约八核心）

每个用户回合（`asr.final` 到达）触发一次循环：

```
loop(turn):
  ctx = 感知(asr.final, 工作记忆, 可用工具清单)
  for round in 0..N(loop_cap):                 # N = 契约值 max_tool_rounds，默认 2
      decision = planner_LLM(ctx)               # deepseek-chat，结构化输出
      if decision.kind == "answer":             # 快路径：无工具，直接作答
          candidate = decision.text; break
      if decision.kind == "tool_calls":
          results = dispatch(decision.tools)     # 受视觉预算 / 重试上限约束
          ctx = ctx + results; continue
      if decision.kind == "clarify":             # 澄清动作（受澄清上限护栏约束）
          candidate = decision.question; break
  else:                                          # 触顶 loop_cap
      candidate = 降级话术(FALLBACK_TEXT)        # 不再无限循环
  reply = 护栏(candidate)                         # 答案拦截 + 置信门控，不可绕过
  send tts.say(reply)
```

要点：
- **快路径优先**：planner 被提示"能直接答就别调工具"。识题后的引导回合应全部走快路径（0 工具）。
- **loop 上限 = 确定性边界**：每回合工具往返 ≤ `max_tool_rounds`（契约值），防失控循环、控延迟与成本。
- **超时即维持现场景**：planner 调用设超时（契约值，默认 800ms 软超时）；超时/失败 → 维持当前 focus、走降级话术，**绝不阻塞首响**（呼应"非确定性不许出现在时序"）。

### 7.3 工具注册表（契约八的一部分，权属归 A）

| 工具 | 入参 | 出参 | MOCK 行为 |
|---|---|---|---|
| `vision.read_problem` | （即刻抓帧） | `{problem_text, confidence}` | `MOCK_VISION=1` 读 fixture |
| `vision.check_draft` | （即刻抓帧） | 四值 verdict `{verdict, error_line?, error_type?, confidence}` | fixture |
| `vision.observe` | `{hint?}` | `{description, confidence}`（通用：穿搭/即兴物体） | fixture |
| `weather.get` | `{lat, lon}` | `{summary, temp, precip, …}` | `MOCK_WEATHER=1` 返回 config 兜底 |
| `memory.note` / `memory.recall` | `{key, value?}` | 工作记忆读写（仅内存） | 进程内 |

说明：
- **"说话"不是工具**，是循环的终点——候选回复穿过护栏 → B 的 `tts.say`。
- **`posture.*` 不进工具表**：姿态是 push（D 发 alert），不是 agent 可拉取的工具，确保 agent 不能把"提醒"变成自由动作。
- **新增 `observe` 是 v0.4 视觉 schema 的开放**：契约三 kind 枚举从三值封死 → 增加 `observe`（见 §7.7 变更清单）。

### 7.4 确定性护栏（设计脊梁的落地，agent 不可覆盖）

| 护栏 | 作用 | 位置 | 契约 |
|---|---|---|---|
| 答案拦截 | 候选回复出站前正则组合拦截 `solved_answer` | 循环之外、TTS 之前 | 九 |
| 置信门控 | 视觉工具 `confidence` 低 → 不播报错误/不开讲，改请用户念/口述 | dispatch 后、作答前 | 三 |
| 澄清上限 | 每 focus 澄清 ≤1 次，超限直接请用户报题号 | planner clarify 动作的硬计数 | 八 |
| loop 上限 | 每回合工具往返 ≤ `max_tool_rounds` | 循环控制 | 八 |
| 视觉预算 | 单会话/单题视觉调用上限（识题1+批改≤2） | dispatch 前 | 八 / C8 |
| 粘滞兜底 | planner 超时/失败 → 维持当前 focus | 循环超时分支 | 八 |
| 姿态不抢话 | 提醒仅在 `gap.open` 插入，可丢弃不排队 | 间隙仲裁，独立于 agent | 四 |

> **判定口径**：粘滞 / 澄清 / 低置信 走**双轨**——planner 策略（提示词层，尽量做对）+ 确定性护栏（代码层，做错也兜住）。**护栏是地板，planner 是天花板**；演示安全只依赖地板。

### 7.5 语音交互体验目标（agentic 化后的延迟硬题）

| 指标 | 目标 | 手段 |
|---|---|---|
| 首响延迟 | 文本回合 ≤ 1.5s；**含视觉/天气工具回合：首句填充语 ≤ 1.5s，实质回答尽力而为** | 快路径（0 工具）= 1 次 LLM 直答；工具回合用复述/填充语（"我看看啊…"）盖住往返；按句 TTS 首句先播；**planner 超时 800ms 维持现场景不阻塞** |
| 工具往返预算 | 每回合 ≤ `max_tool_rounds`（默认 2） | loop 上限护栏；多数引导回合 0 工具 |
| 打断支持 | 双模式正式特性 | **自由对话**：VAD 打断（AEC + SPEAKING 态 ≥300ms 连续人声才判打断）；**对讲机模式（正式特性）**：按住空格说话、按键打断，一键无缝切换（不刷新、TTS 不中断）。开场即介绍两模式 |
| 轮次判定 | 不抢话、不冷场 | 纯 VAD 静默阈值；对讲机模式由按键确定 |

> **诚实记录**：工具调用回合的实质回答延迟天然高于 v0.4 单次调用，靠"复述确认 + 填充语 + 快路径占多数"对冲。彩排须专测"识题/批改回合"的延迟观感。

### 7.6 成本控制策略清单

| # | 策略 | 适用 | 状态 |
|---|---|---|---|
| C1 | 姿态/VAD 全端侧，零云调用 | S2 全程，触发判断 | **采用** |
| C2 | 识题结果文本驻留 + **工具自由快路径**（引导回合 0 工具，纯文本模型） | S1（最大杠杆） | **采用（强化）** |

## 8. 降级与稳定性策略（演示保障）

| 风险 | 降级方案 |
|---|---|
| **agent 循环不稳定 / 乱跑**（agentic 新增主风险） | **M2 硬门 → rails mode**：锁定确定性工具序（识题→引导→批改定序，agent 只填语言不自主选工具）；rails mode 本身 M4 须彩排一次（见 §10 止损线） |
| agent 失控循环 / 超调用 | loop 上限 + 视觉预算护栏（C8）；触顶即 FALLBACK_TEXT |
| 现场网络弱 | 核心链路预热缓存；S1 备用题预识别；天气写死兜底；TTS 失败回退屏幕字幕 |
| 现场噪声 VAD 误触发/自激 | 一键切对讲机模式（正式特性，有主持词包装）；间隙判定自动切定时开窗 |
| 摄像头/麦克风权限被拒 | 引导文案重试；终极降级为纯文字模式 |
| 识题失败 | 澄清 ≤1 次 → 请用户报题号/口述 |
| 批改低置信 | 四值 verdict 门控：不播报错误，改请学生念该行；彩排预授权口述批改剧本 |
| ASR 误识 | 复述确认兜底；歧义即澄清；关键操作先确认再执行 |
| 开放面（S-OPEN）翻车 | 置信门控强制"看不清/帮不上就明说"，绝不编造 |

---

## 9. 演示动线（5–6 分钟 + 备选挑战环节）

1. **开场（30s）**：学生坐下唤起助手，一句话产品定位（"一个自己会编排、又管得住自己的学习伙伴"）+ **介绍两种交互模式**（自由对话/对讲机）。
2. **主场景（3–4min）**：指题求助 →（agent 自主识题）复述确认 → 苏格拉底引导（agent 临场决定粒度，多数回合零工具走快路径）→ 写完请批改（agent 自主抓帧，预埋移项符号错误）→ 中途自然触发一次驼背提醒（间隙插入，展示"不打扰"且"提醒非 agent 自发"）。
3. **生活串场（45s）**：学生说"我要出门了" → 照镜头问穿搭（agent 自主编排 `observe` + `weather.get`）。
4. **收尾（30s）**：说"今天就到这" → AI 口头小结 + 坐姿提醒次数；同屏成本看板（标"估算值"，可口播"今天大多数回合没调云端"）。
5. **备选：开放挑战环节（S-OPEN，时间/氛围允许时主动发起）**：邀评委从练习册任选一题、当场手写一题、或问画面里任意物体（坦诚话术："即兴成功率不保证，敢不敢试"）。防线：置信门控强制"看不清就明说、请口述"，绝不编造——**这一环正是 agentic 自主编排的最佳展示位**。

## 10. 里程碑（1 人 × 36h，多 agent 并行版，含 agentic 止损线）

| 阶段 | 时间 | 本人（串行） | AI agent（并行） | 冒烟点 / 止损线 |
|---|---|---|---|---|
| M0 契约 | 0–1.5h | 写契约（一~十）+ 工具注册表 schema + planner prompt + fixture；**冻结契约八（agent 循环）与契约十** | — | 契约入库 |
| M1 语音链路 | 1.5–10h | 真机调 VAD/流式 ASR/按句 TTS/打断时序 | 四路并行：A=agent 编排器骨架（loop + dispatch + 护栏 + loop 上限）/ C=视觉工具 / D=姿态 / E=技能提示词（**每 2h 本人验收一次**） | **第 4h**：裸链路（说话→ASR 回显）不通 → 放弃 VAD 打断，全程对讲机；**第 7h**：回声机器人 10 次 <8 达 → 打断锁按键；**第 10h 硬墙**：锁定最稳语音模式，禁碰底层 |
| M2 主场景合体 | 10–18h | 集成：mock 工具跑通 agent 循环（识题→引导→批改）→ 真工具接入 + 答案拦截 + 置信门控 | E 提示词打磨、答案拦截/护栏测试、planner 工具选择调参 | 主场景跑通；**agentic 硬门**：agent 循环连续 3 次干跑不能稳定驱动 S1 → **锁 rails mode**（确定性工具序，agent 只填语言）；**落后 >2h → 砍 S3** |
| M3 守护+插件+开放 | 18–26h | 集成姿态守护（间隙提醒不抢话）；S3（`observe`+`weather`）；S-OPEN 开放挑战路径 + 诚实兜底 | S3 编排、成本看板、家长摘要（零状态）、`observe` fixture | 提醒段 + 串场段 + 开放挑战段跑通 |
| M4 加固 | 26–30h | 降级全量验证：rails mode 切换、对讲机无缝、口述兜底、断网兜底、答案拦截误杀专测、loop 上限/失控专测 | — | 任一降级路径可演；**rails mode 彩排过一次** |
| M5 彩排 | 30–36h | **彩排 ×3 + buffer（神圣不可侵犯）**：全练习册逐题、陌生字迹挑战 ≥3、模糊草稿验 unreadable、三次"我不会"压力测试、噪声一轮、**agent 循环一致性专测（同动线 3 次不乱跑）** | — | 全动线连续 3 次无致命失误 |

> **工时账与 agentic 风险对冲**：名义工时与上一线同量级（砍 S4 回收、agent 并行消化纯逻辑模块、产出 review 按 20% 计提）。**新增风险**：用 agent 循环替换确定性路由，集成与调参不确定性高于 v0.4。**对冲**：① M2 硬门 + rails mode 作为 agentic 专属止损（对标语音的 4h/7h/10h）；② 护栏全确定性，把非确定性圈在"语言/策略"层；③ S3 仍是唯一牺牲位。**分歧记录在案**：决策人在知情 agentic 抬高彩排难度的前提下仍选定本路线，rails mode 即为该分歧的兜底机制。

## 11. 成功指标（验收口径）

- **视觉准确性**：指题识别一次命中率 ≥ 80%（整本练习册实测）；澄清 ≤1 次；批改在预埋错误上定位正确率 100%；模糊草稿输出 unreadable 而非编造。
- **编排自主性（本版新增）**：S1 全程 agent 自主决定识题/引导/批改时机，无写死序列；**同一演示动线连续 3 次行为一致、不乱跑、不超 loop 上限**；开放挑战（S-OPEN）≥3 次实测中，**不编造**（看不清即明说）为硬指标，"答对"为软指标。
- **语音自然度**：文本回合首响 ≤1.5s；工具回合首句填充 ≤1.5s；打断可用（VAD 或按键任一）；对讲机切换无缝；全程无"抢话/死冷场"。
- **成本控制**：S2 云调用 = 0；S1 单题视觉 ≤ 3 帧；S3 ≤ 1 帧；**引导回合多数走 0 工具快路径**；全程花费可量化（标"估算值"）。
- **产品立场**：全程 AI 不直接给出任何一道题的最终答案；**连续三次"我不会"不破防**（答案拦截硬防线兜底）；开放面失败时优雅承认而非编造。

## 附录 A：术语表

| 术语 | 说明 |
|---|---|
| Agent 编排循环 | 反应式工具调用循环：感知→规划→（调工具）→作答，受 loop 上限约束（§7.2） |
| 工具注册表 | agent 可调用的工具集（`vision.*`/`weather.get`/`memory.*`），权属归 A（§7.3） |
| 工作记忆 | 会话内仅内存的状态（active_problem/solved_answer/focus/…），契约十 |
| 确定性护栏 | 编排循环之外、agent 不可覆盖的硬约束（答案拦截/置信门控/澄清上限/loop 上限）（§7.4） |
| 快路径 | planner 直接出文本、零工具调用的回合，引导对话主力，最省成本 |
| rails mode | agentic 止损兜底：锁定确定性工具序，agent 只填语言（§8/§10） |
| 反应式 / 主动发起 | 反应式 = 被动等用户回合（本版决策）；主动发起 = agent 自发开口（本版**不做**，姿态提醒是独立确定性护栏，非 agent 自发） |
| VAD | Voice Activity Detection，语音活动检测 |
| barge-in | 用户打断 AI 播报，AI 立即停播转入聆听 |
| 对讲机模式 | push-to-talk，正式双模式之一 |
| 苏格拉底式引导 | 通过连续提问引导学生自行推导，而非直接给答案 |
